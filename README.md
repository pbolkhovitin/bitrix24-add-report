# SLA-отчёт техподдержки Bitrix24

Веб-дашборд для отслеживания SLA службы технической поддержки,
работающей в **Bitrix24 on-premise** через сущность **Задачи (Tasks)**.

## Возможности

- 📊 KPI-карточки: процент в SLA, среднее время первого ответа и решения
- 📈 Графики: динамика обращений, распределение по статусам и исполнителям
- 🔍 Фильтры: дата, исполнитель, приоритет, группа, сигнал первого ответа
- 📋 Таблица задач с сортировкой и цветовой индикацией SLA
- ⚙️ Настройка рабочих часов и порогов SLA через UI
- 🧪 Демо-режим с 40 примерными обращениями (без живого портала)
- 🔄 Автоматическая синхронизация с Bitrix24 по расписанию

## Требования

- Python 3.11+
- Bitrix24 on-premise с доступом к REST API
- Права администратора для создания входящего вебхука

## Установка

```bash
# Клонировать репозиторий

```

## Настройка

1. Создать входящий вебхук в Bitrix24:
   - Перейти: **Разработчикам → Входящий вебхук → Добавить вебхук**
   - Выбрать права: `task` (задачи) и `user` (пользователи)
   - Скопировать URL вида `https://ваш-портал/rest/1/ABCDE123456/`
     (`1` — это `webhook_user`, `ABCDE123456` — `webhook_token`)

2. Найти ID пользователя «Техническая поддержка» (`tp_user_id`):
   - Открыть профиль пользователя в портале; ID есть в URL (`user/<ID>/`),
     либо запросить через REST `user.get` с фильтром по имени, либо в таблице `b_user`.

3. Создать в корне проекта файл `config.yaml` (без него приложение
   стартует в демо-режиме) и заполнить по шаблону ниже:

```yaml
bitrix:
  portal_url: "https://ваш-портал.bitrix24.ru"   # без завершающего слэша
  webhook_user: "1"            # число из URL вебхука (ID пользователя-владельца)
  webhook_token: "PASTE_WEBHOOK_CODE"  # код из URL вебхука
  tp_user_id: 1               # RESPONSIBLE_ID служебного пользователя «Техподдержка»
  request_timeout: 30
  batch_size: 50

sync:
  interval_minutes: 60        # период автосинхронизации с порталом

# Рабочее время — SLA-таймеры считаются только в этих окнах.
# workdays: 0=Вс,1=Пн,2=Вт,3=Ср,4=Чт,5=Пт,6=Сб
# holidays: список дат YYYY-MM-DD (вручную; в будущем — из календаря Bitrix24).
working_hours:
  start: "09:00"
  end: "18:00"
  workdays: [1, 2, 3, 4, 5]
  holidays: []
  # Пример: ["2026-01-01", "2026-01-02", "2026-01-07", "2026-03-08", "2026-05-01"]

# Нормативы SLA по приоритету задачи (PRIORITY), в минутах бизнес-времени.
# Ключ "default" — если приоритет не указан явно.
# Приоритеты Bitrix24: 1 — низкий, 2 — обычный, 3 — высокий, 4 — критичный.
sla:
  thresholds:
    default: { first_response_minutes: 120, resolution_minutes: 480 }
    1: { first_response_minutes: 480, resolution_minutes: 1440 }   # низкий
    2: { first_response_minutes: 240, resolution_minutes: 960  }   # обычный
    3: { first_response_minutes: 60,  resolution_minutes: 480 }    # высокий
    4: { first_response_minutes: 15,  resolution_minutes: 240 }    # критичный

server:
  host: "0.0.0.0"
  port: 8080
```

4. Рабочие часы и пороги SLA можно позже править прямо в UI дашборда
   (кнопка «Настройки») — изменения сохраняются в `sla_config_override.json`.

## Запуск

### Демо-режим (без Bitrix24)

```bash
python -m sla.app --demo
```

Откройте http://localhost:8080

### Рабочий режим

```bash
python -m sla.app
```

С указанием порта:
```bash
python -m sla.app --port 9090
```

С переменными окружения:
```bash
SLA_DEMO=1 python -m sla.app
SLA_CONFIG=/path/to/config.yaml python -m sla.app
```

## Как считается SLA

1. **Рабочее время**: SLA-таймеры считаются только в рабочие часы (по умолчанию
   Пн–Пт, 09:00–18:00, без учёта праздников). Время вне этого окна не засчитывается.

2. **Первый ответ**: отсчитывается от создания задачи до момента первого действия
   исполнителя. В интерфейсе можно выбрать сигнал:
   - **Смена статуса** (рекомендуется) — первое изменение статуса задачи не создателем
   - **Комментарий** — первый комментарий от исполнителя
   - **Учёт времени** — первая запись затраченного времени

3. **Решение**: от создания задачи до закрытия (статус «Завершена»).

4. **Пороги SLA** задаются отдельно для каждого приоритета (и «по умолчанию»)
   в минутах бизнес-времени.

## Troubleshooting

- **Комментарии не загружаются**: на новых версиях Bitrix24 комментарии задач
  перенесены в Живую ленту. Используйте сигнал «Смена статуса» как основной.
- **Ошибка прав вебхука**: убедитесь, что вебхуку выданы права `task` и `user`.
- **Медленная синхронизация**: уменьшите `batch_size` в конфиге или увеличьте
  `request_timeout`.
- **Лимиты API**: on-premise версии обычно не имеют строгих лимитов, но при
  большом количестве задач (~1000+) синхронизация может занимать несколько минут.
