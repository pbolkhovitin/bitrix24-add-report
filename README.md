# SLA-отчёт техподдержки Bitrix24

Веб-дашборд для отслеживания SLA службы технической поддержки,
работающей в **Bitrix24 on-premise** через сущность **Задачи (Tasks)**.

## Возможности

- 📊 KPI-карточки: процент в SLA, среднее время первого ответа и решения
- 📈 Графики: динамика обращений, распределение по статусам и исполнителям
- 🔍 Фильтры: дата, исполнитель, приоритет, группа, сигнал первого ответа
- 📋 Таблица задач с сортировкой и цветовой индикацией SLA
- ⚙️ Настройка рабочих часов и порогов SLA через UI
- 🧪 Демо-режим с 40 примерными обращениями (без живого портала)
- 🔄 Автоматическая синхронизация с Bitrix24 по расписанию
- ⚠️ Отчёт отклонений (8 типов: без описания результата, без учёта времени, зависшие задачи, без исполнителя, лишние соисполнители, перенос дедлайна без комментария, незаполненные обязательные поля, многокорпоративные без родителя)
- 📋 Контроль диспетчера — ежедневный чек-лист
- 📅 Еженедельный разбор — задачи >7 дней, просроченные, без времени/результата
- 🔀 Анализ маршрутизации — счётчики передач, эскалаций, возвратов
- 📑 Вкладки: Дашборд, Отклонения, Контроль диспетчера, Еженедельный разбор, Маршрутизация

## Требования

- Python 3.11+
- Bitrix24 on-premise с доступом к REST API
- Права администратора для создания входящего вебхука

## Настройка

1. Создать входящий вебхук в Bitrix24:
   - Перейти: **Разработчикам → Входящий вебхук → Добавить вебхук**
   - Выбрать права: `task` (задачи) и `user` (пользователи)
   - Скопировать URL вида `https://ваш-портал/rest/1/ABCDE123456/`
     (`1` — это `webhook_user`, `ABCDE123456` — `webhook_token`)

2. Найти ID пользователя «Техническая поддержка» (`tp_user_id`):
   - Открыть профиль пользователя в портале; ID есть в URL (`user/<ID>/`),
     либо запросить через REST `user.get` с фильтром по имени, либо в таблице `b_user`.

3. Создать в корне проекта файл `config.yaml` (без него приложение
   стартует в демо-режиме) и заполнить по шаблону ниже:

```yaml
bitrix:
  portal_url: "https://ваш-портал.bitrix24.ru"   # без завершающего слэша
  webhook_user: "1"            # число из URL вебхука (ID пользователя-владельца)
  webhook_token: "PASTE_WEBHOOK_CODE"  # код из URL вебхука
  tp_user_id: 1               # RESPONSIBLE_ID служебного пользователя «Техподдержка»
  request_timeout: 30
  batch_size: 50

sync:
  interval_minutes: 60        # период автосинхронизации с порталом

# Рабочее время — SLA-таймеры считаются только в этих окнах.
# workdays: 0=Вс,1=Пн,2=Вт,3=Ср,4=Чт,5=Пт,6=Сб
# holidays: список дат YYYY-MM-DD (вручную; в будущем — из календаря Bitrix24).
working_hours:
  start: "09:00"
  end: "18:00"
  workdays: [1, 2, 3, 4, 5]
  holidays: []
  # Пример: ["2026-01-01", "2026-01-02", "2026-01-07", "2026-03-08", "2026-05-01"]

# Нормативы SLA по приоритету задачи (PRIORITY), в минутах бизнес-времени.
# Ключ "default" — если приоритет не указан явно.
# Приоритеты Bitrix24: 1 — низкий, 2 — обычный, 3 — высокий, 4 — критичный.
sla:
  thresholds:
    default: { first_response_minutes: 120, resolution_minutes: 480 }
    1: { first_response_minutes: 480, resolution_minutes: 1440 }   # низкий
    2: { first_response_minutes: 240, resolution_minutes: 960  }   # обычный
    3: { first_response_minutes: 60,  resolution_minutes: 480 }    # высокий
    4: { first_response_minutes: 15,  resolution_minutes: 240 }    # критичный

# Настройки процесса техподдержки
process:
  deviations:
    stuck_days_threshold: 2          # дни без смены статуса → «зависшая»
    comment_window_minutes: 60       # окно вокруг переноса дедлайна для поиска комментария
    min_result_comment_length: 20    # мин. длина комментария-результата
  stages:
    operator_minutes: 30             # норматив на стадии оператора
    dispatcher_minutes: 120          # норматив на стадии диспетчера
    head_minutes: 360                # норматив на стадии руководителя
  mandatory_fields:
    std_fields: ["DESCRIPTION"]      # стандартные поля, обязательные к заполнению
    uf_fields: []                    # пользовательские UF_* поля (если есть)
  status_mapping: {}                 # маппинг статус→метка (пусто = стандартные)
  user_roles: {}                     # маппинг user_id→роль (например {"1":"operator"})

server:
  host: "0.0.0.0"
  port: 8080
```

4. Рабочие часы и пороги SLA можно позже править прямо в UI дашборда
   (кнопка «Настройки») — изменения сохраняются в `sla_config_override.json`.

## Подключение к рабочему Bitrix24

### Шаг 1. Создать входящий вебхук

В портале Bitrix24:

1. Перейти: **Разработчикам → Входящий вебхук → Добавить вебхук**
2. Выбрать права: **task** (задачи) и **user** (пользователи)
3. Скопировать URL вида `https://ваш-портал/rest/1/ABCDE123456/`
   - `1` — это `webhook_user` (ID пользователя-владельца вебхука)
   - `ABCDE123456` — это `webhook_token` (код доступа)

### Шаг 2. Найти ID пользователя «Техническая поддержка»

Это `tp_user_id` — по нему фильтруются задачи службы поддержки
(поле `RESPONSIBLE_ID` в Bitrix24). Способы найти:

- Открыть профиль пользователя в портале — ID есть в URL (`user/<ID>/`)
- Запросить через REST: `user.get` с фильтром по имени
- Найти в таблице `b_user` базы данных портала

### Шаг 3. Проверить соединение

Выполнить запрос и убедиться, что возвращается JSON с данными пользователя:

```bash
curl "https://ваш-портал/rest/1/ABCDE123456/user.get?ID=1"
```

Если ответ содержит `result` с полями пользователя — вебхук работает.

### Шаг 4. Создать config.yaml

Скопировать шаблон из раздела [Настройка](#настройка) выше, заполнить
`portal_url`, `webhook_user`, `webhook_token`, `tp_user_id` реальными
значениями и сохранить в корне проекта.

Файл `config.yaml` добавлен в `.gitignore` — токен не попадёт в репозиторий.

### Шаг 5. Запустить

```bash
.venv/bin/python -m sla.app
```

Без флага `--demo` и с заполненным `config.yaml` приложение подключится
к порталу и начнёт синхронизацию задач. Первый запуск загрузит все задачи
пользователя «Техподдержка», далее — автообновление по расписанию
(`sync.interval_minutes`, по умолчанию 60 минут).

Пороги SLA и рабочие часы можно скорректировать позже через UI дашборда
(кнопка «Настройки») — изменения сохраняются в `sla_config_override.json`.

> 📖 Подробное пошаговое руководство со всеми нюансами коробочной версии — см. **[BITRIX24_SETUP.md](BITRIX24_SETUP.md)**.

## Как считается SLA

1. **Рабочее время**: SLA-таймеры считаются только в рабочие часы (по умолчанию
   Пн–Пт, 09:00–18:00, без учёта праздников). Время вне этого окна не засчитывается.

2. **Первый ответ**: отсчитывается от создания задачи до момента первого действия
   исполнителя. В интерфейсе можно выбрать сигнал:
   - **Смена статуса** (рекомендуется) — первое изменение статуса задачи не создателем
   - **Комментарий** — первый комментарий от исполнителя
   - **Учёт времени** — первая запись затраченного времени

3. **Решение**: от создания задачи до закрытия (статус «Завершена»).

4. **Пороги SLA** задаются отдельно для каждого приоритета (и «по умолчанию»)
   в минутах бизнес-времени.

## Отчёт отклонений и контроль процесса

Помимо базовых SLA-метрик, дашборд выявляет отклонения в процессе техподдержки:

### Типы отклонений (8 типов)

| Тип | Уровень | Описание |
|-----|---------|----------|
| `no_result_description` | высокий | Закрыта без комментария-результата (или комментарий < 20 симв.) |
| `no_time_logged` | высокий | Закрыта без учёта затраченного времени |
| `no_executor_assigned` | высокий | Исполнитель не назначен (RESPONSIBLE_ID == CREATED_BY) |
| `stuck_task` | средний | Нет смены статуса > N дней (по умолчанию 2) |
| `deadline_moved_without_comment` | средний | Дедлайн перенесён без комментария в окне ±60 мин |
| `missing_mandatory_fields` | средний | Не заполнены обязательные поля (DESCRIPTION, UF_*) |
| `multi_department_without_parent` | низкий | Смена группы >2 раз без родительской задачи |
| `unnecessary_accomplices` | низкий | Есть соисполнители (часто избыточны) |

### Вкладки дашборда

- **Дашборд** — KPI, графики, таблица задач
- **Отклонения** — все отклонения с фильтром по уровню и типу
- **Контроль диспетчера** — ежедневный чек-лист: новые без обработки, без обязательных полей, без владельца, без результата, без времени, зависшие
- **Еженедельный разбор** — задачи >7 дней, просроченные, без времени, без результата, переносы дедлайна
- **Маршрутизация** — счётчики передач/эскалаций/возвратов по задачам

### Настройка процесса

Раздел `process` в `config.yaml` (см. шаблон ниже) позволяет настроить:
- Пороги обнаружения отклонений (`deviations`)
- Нормативы по стадиям (`stages`)
- Обязательные поля (`mandatory_fields`)
- Маппинг статусов и ролей (`status_mapping`, `user_roles`)

## Troubleshooting

- **Комментарии не загружаются**: на новых версиях Bitrix24 комментарии задач
  перенесены в Живую ленту. Используйте сигнал «Смена статуса» как основной.
- **Ошибка прав вебхука**: убедитесь, что вебхуку выданы права `task` и `user`.
- **Медленная синхронизация**: уменьшите `batch_size` в конфиге или увеличьте
  `request_timeout`.
- **Лимиты API**: on-premise версии обычно не имеют строгих лимитов, но при
  большом количестве задач (~1000+) синхронизация может занимать несколько минут.
- **Отклонения не показываются**: убедитесь, что задачи синхронизированы (раздел «Дашборд» → кнопка «Синхронизировать»). В демо-режиме отклонения генерируются автоматически.
- **Статусы отображаются числами**: заполните `process.status_mapping` в config.yaml (например `{"1":"Новая", "3":"В работе"}`).
- **Роли пользователей не определены**: заполните `process.user_roles` (например `{"1":"operator", "2":"dispatcher"}`).
