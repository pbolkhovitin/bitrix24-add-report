## Требования к порталу

- **Коробочная версия Bitrix24** — любая редакция с REST API: Стандарт, Профессиональ, Корпоратив, Энтерпрайз.
- **Доступ администратора** к порталу — для создания входящего вебхука и управления пользователями.
- **REST API включён** — в коробочной версии REST API активен по умолчанию, дополнительная настройка не требуется.
- **Сетевой доступ** — сервер, на котором запущен дашборд, должен иметь HTTPS-доступ к порталу (порт 443). Если портал за NAT/firewall — откройте доступ или используйте туннель.
- **Python 3.11+** на сервере дашборда.

## Подготовка служебного пользователя

**Зачем нужен:** дашборд фильтрует задачи по `RESPONSIBLE_ID` — все обращения техподдержки должны быть назначены на одного служебного пользователя.

**Как создать:**

1. Перейти: **Администрирование → Пользователи → Добавить пользователя**
2. Заполнить поля:
   - **Имя**: «Техническая поддержка» (или любое другое, на ваше усмотрение)
   - **Email**: можно фиктивный, например `tech-support@ваш-портал`
   - **Логин**: tech_support (не используется дашбордом, но пригодится)
   - **Роль**: любая (дашборд не использует роли Bitrix24 — только `RESPONSIBLE_ID`)
   - **Активность**: активен
3. Сохранить пользователя
4. **Записать ID пользователя** — он виден в URL профиля: `https://ваш-портал/company/personal/user/42/` → ID = `42`

**Важно:** этот пользователь должен быть указан как `RESPONSIBLE_ID` для всех задач техподдержки. Если задачи создаются на других исполнителей — они **не попадут** в дашборд. Решения:
- Назначать все задачи ТП на этого служебного пользователя (рекомендуется)
- Изменить логику фильтрации в `sla/sync.py` (параметр `tp_user_id`)
- Использовать несколько `tp_user_id` (пока не реализовано — потребуется доработка)

## Создание группы/проекта техподдержки

**Зачем:** задачи техподдержки обычно находятся в отдельной группе. Дашборд может фильтровать по `GROUP_ID`.

**Как создать:**

1. Перейти: **Задачи и проекты → Группы проектов → Создать группу**
2. Заполнить:
   - **Название**: «Техническая поддержка» (или любое)
   - **Тип**: закрытая или открытая (не влияет на работу REST API)
3. Сохранить группу
4. **Записать ID группы** — виден в URL: `https://ваш-портал/workgroups/group/12/` → ID = `12`

**Использование:** ID группы можно передавать в фильтр дашборда (поле «Группа» в UI). Фильтрация по группе реализована на уровне API (`filter[GROUP_ID]`).

## Настройка задач техподдержки

**Стандартные поля Bitrix24 для задач:**

| Поле | Описание | Тип |
|------|----------|-----|
| `TITLE` | Название задачи | строка |
| `DESCRIPTION` | Описание задачи | строка (HTML) |
| `RESPONSIBLE_ID` | Ответственный исполнитель | число (user ID) |
| `CREATED_BY` | Постановщик | число (user ID) |
| `DEADLINE` | Крайний срок | datetime |
| `PRIORITY` | Приоритет (1–4) | число |
| `STATUS` | Статус (1–7) | число |
| `GROUP_ID` | Группа проекта | число |
| `ACCOMPLICES` | Соисполнители | JSON-массив ID |
| `AUDITORS` | Наблюдатели | JSON-массив ID |
| `TIME_ESTIMATE` | Плановая трудоёмкость (сек) | число |
| `DURATION_FACT_SECONDS` | Фактическая длительность (сек) | число |
| `MARK` | Оценка (0=нет, 2=положит., 3=отрицат.) | число |
| `PARENT_ID` | Родительская задача | число |
| `TAGS` | Теги | JSON-массив |

**Пользовательские поля (UF_\*):** создаются в портале вручную, начинаются с `UF_`. Дашборд запрашивает все UF_\* поля через `select[]=*&select[]=UF_*`.

**Статусы задач (стандартные):**

| Код | Статус |
|-----|--------|
| 1 | Новая |
| 2 | Ждёт выполнения |
| 3 | В работе |
| 4 | Ожидает |
| 5 | Завершена |
| 6 | Отложена |
| 7 | Отклонена |

В коробочной версии администратор может создать кастомные статусы. В этом случае заполните `process.status_mapping` (см. раздел 11).

**Рекомендация:** заведите единый шаблон задачи техподдержки с обязательными полями (как минимум `TITLE`, `DESCRIPTION`, `RESPONSIBLE_ID`).

## Создание входящего вебхука

**Подробная инструкция:**

1. Войдите в портал под **администратором**
2. Откройте левое меню → **Разработчикам** (прямой URL: `https://ваш-портал/dev/`)
3. Выберите раздел **Входящий вебхук** → **Добавить вебхук**
4. Заполните форму:
   - **Название**: «SLA Dashboard» (любое)
   - **URL обработчика**: оставьте пустым (для входящего вебхука не нужен)
   - **Права доступа** — отметьте **только** необходимые:
     - ✅ **Задачи (task)** — обязательно
     - ✅ **Пользователи (user)** — обязательно
     - ❌ Чат и уведомления — не нужно
     - ❌ CRM — не нужно (если только не используете CRM-связь; дашборд её не требует)
5. Нажмите **Сохранить**

После сохранения появится URL вида:
```
https://ваш-портал/rest/1/AbCdEf123456/
```

- `1` — это `webhook_user` (ID пользователя-владельца вебхука)
- `AbCdEf123456` — это `webhook_token` (код доступа)

**Скопируйте URL** — он понадобится в config.yaml.

**Важно:** вебхук работает от имени пользователя-владельца. У этого пользователя должны быть права на просмотр **всех** задач техподдержки (иначе REST вернёт только его личные задачи). Рекомендуется создавать вебхук от имени администратора.

**Безопасность:** URL вебхука — это пароль. Не коммитьте его в git, не публикуйте. `config.yaml` добавлен в `.gitignore`.

## Поиск ID пользователя «Техподдержка»

Три способа:

### Способ 1 — через портал (проще всего)

Откройте профиль пользователя «Техническая поддержка». ID есть в URL:
```
https://ваш-портал/company/personal/user/42/
```
→ `tp_user_id = 42`

### Способ 2 — через REST API

```bash
# Все пользователи
curl -s "https://ваш-портал/rest/1/AbCdEf123456/user.get" | python -m json.tool

# Фильтр по имени
curl -s "https://ваш-портал/rest/1/AbCdEf123456/user.get?FILTER[NAME]=Техническая" | python -m json.tool
```

В ответе найдите поле `ID` нужного пользователя.

### Способ 3 — через базу данных портала (коробка)

Если есть доступ к БД MySQL/MariaDB портала:

```sql
SELECT ID, NAME, LAST_NAME FROM b_user WHERE NAME LIKE '%Техническая%';
```

Параметры подключения к БД — в файле `/bitrix/php_interface/dbconn.php`.

## Проверка соединения с REST API

Выполните 4 проверки, чтобы убедиться, что вебхук работает и имеет нужные права:

```bash
# 1. Проверка прав user — получить пользователя с ID=1
curl -s "https://ваш-портал/rest/1/AbCdEf123456/user.get?ID=1" | python -m json.tool

# 2. Проверка прав task — список задач (заменить 42 на tp_user_id)
curl -s "https://ваш-портал/rest/1/AbCdEf123456/tasks.task.list?filter[RESPONSIBLE_ID]=42&start=0" | python -m json.tool

# 3. История задачи (заменить TASK_ID на реальный ID из п.2)
curl -s "https://ваш-портал/rest/1/AbCdEf123456/tasks.task.history.list?taskId=123" | python -m json.tool

# 4. Учёт времени по задаче
curl -s "https://ваш-портал/rest/1/AbCdEf123456/task.elapseditem.getlist?TASKID=123" | python -m json.tool
```

**Ожидаемый ответ:** JSON с полем `result`. Если ответ содержит `error` — проверьте права вебхука (раздел 5).

## Создание config.yaml

Создайте в корне проекта файл `config.yaml`. **Полный шаблон** со всеми секциями:

```yaml
bitrix:
  portal_url: "https://ваш-портал.bitrix24.ru"   # без завершающего слэша
  webhook_user: "1"            # число из URL вебхука (ID пользователя-владельца)
  webhook_token: "PASTE_WEBHOOK_CODE"  # код из URL вебхука
  tp_user_id: 42              # RESPONSIBLE_ID служебного пользователя «Техподдержка»
  request_timeout: 30
  batch_size: 50

sync:
  interval_minutes: 60        # период автосинхронизации с порталом

# Рабочее время — SLA-таймеры считаются только в этих окнах.
# workdays: 0=Вс,1=Пн,2=Вт,3=Ср,4=Чт,5=Пт,6=Сб
# holidays: список дат YYYY-MM-DD (вручную; в будущем — из календаря Bitrix24).
working_hours:
  start: "09:00"
  end: "18:00"
  workdays: [1, 2, 3, 4, 5]
  holidays: []
  # Пример: ["2026-01-01", "2026-01-02", "2026-01-07", "2026-03-08", "2026-05-01"]

# Нормативы SLA по приоритету задачи (PRIORITY), в минутах бизнес-времени.
# Ключ "default" — если приоритет не указан явно.
# Приоритеты Bitrix24: 1 — низкий, 2 — обычный, 3 — высокий, 4 — критичный.
sla:
  thresholds:
    default: { first_response_minutes: 120, resolution_minutes: 480 }
    1: { first_response_minutes: 480, resolution_minutes: 1440 }   # низкий
    2: { first_response_minutes: 240, resolution_minutes: 960  }   # обычный
    3: { first_response_minutes: 60,  resolution_minutes: 480 }    # высокий
    4: { first_response_minutes: 15,  resolution_minutes: 240 }    # критичный

# Настройки процесса техподдержки
process:
  deviations:
    stuck_days_threshold: 2          # дни без смены статуса → «зависшая»
    comment_window_minutes: 60       # окно вокруг переноса дедлайна для поиска комментария
    min_result_comment_length: 20    # мин. длина комментария-результата
  stages:
    operator_minutes: 30             # норматив на стадии оператора
    dispatcher_minutes: 120          # норматив на стадии диспетчера
    head_minutes: 360                # норматив на стадии руководителя
  mandatory_fields:
    std_fields: ["DESCRIPTION"]      # стандартные поля, обязательные к заполнению
    uf_fields: []                    # пользовательские UF_* поля (если есть)
  status_mapping: {}                 # маппинг статус→метка (пусто = стандартные)
  user_roles: {}                     # маппинг user_id→роль (например {"1":"operator"})

server:
  host: "0.0.0.0"
  port: 8080
```

Заполните `portal_url`, `webhook_user`, `webhook_token`, `tp_user_id` реальными значениями.

**Примечание:** если файл `config.yaml` отсутствует — приложение стартует в демо-режиме. Для подключения к реальному порталу файл обязателен.

## Настройка рабочих часов и праздников

SLA-таймеры (первый ответ, решение) считаются **только** в рабочие часы. Время вне этого окна не засчитывается.

```yaml
working_hours:
  start: "09:00"             # начало рабочего дня
  end: "18:00"               # конец рабочего дня
  workdays: [1, 2, 3, 4, 5] # 0=Вс, 1=Пн, 2=Вт, 3=Ср, 4=Чт, 5=Пт, 6=Сб
  holidays: []
```

**Параметры:**
- `workdays` — дни недели, в которые работает техподдержка. По умолчанию Пн–Пт (1–5). Для 24/7 укажите `[0,1,2,3,4,5,6]`.
- `holidays` — список праздничных дат в формате `YYYY-MM-DD`. Заполняется вручную. В будущих версиях планируется интеграция с календарём Bitrix24.

**Пример для РФ (2026):**
```yaml
holidays:
  - "2026-01-01"
  - "2026-01-02"
  - "2026-01-03"
  - "2026-01-06"
  - "2026-01-07"
  - "2026-02-23"
  - "2026-03-08"
  - "2026-05-01"
  - "2026-05-09"
  - "2026-06-12"
  - "2026-11-04"
```

**Влияние на метрики:**
- SLA-таймеры (первый ответ, решение) — считаются **только** в бизнес-часы
- Время в статусе (`time_in_status`) — тоже в бизнес-часах
- Дедлайн (`deadline_met`) — по календарному времени (не бизнес-часам)

## Настройка SLA-порогов

Пороги задаются отдельно для каждого приоритета. Указываются в минутах бизнес-времени.

```yaml
sla:
  thresholds:
    default: { first_response_minutes: 120, resolution_minutes: 480 }
    1: { first_response_minutes: 480, resolution_minutes: 1440 }   # низкий
    2: { first_response_minutes: 240, resolution_minutes: 960  }   # обычный
    3: { first_response_minutes: 60,  resolution_minutes: 480 }    # высокий
    4: { first_response_minutes: 15,  resolution_minutes: 240 }    # критичный
```

**Примеры из практики:**

| Приоритет | Первый ответ | Решение |
|-----------|-------------|---------|
| Критичный (P4) | 15 мин | 4 часа |
| Высокий (P3) | 1 час | 8 часов |
| Обычный (P2) | 4 часа | 16 часов |
| Низкий (P1) | 8 часов | 24 часа |

**Изменение через UI:** пороги можно редактировать в дашборде (кнопка «Настройки»). Изменения сохраняются в `sla_config_override.json` (не перезаписывает `config.yaml`).

## Настройка маппинга статусов

**Если используете стандартные статусы Bitrix24** — оставьте `status_mapping: {}` (пусто). Дашборд использует встроенные метки:
- 1 → Новая, 2 → Ждёт выполнения, 3 → В работе, 4 → Ожидает, 5 → Завершена, 6 → Отложена, 7 → Отклонена

**Если создали кастомные статусы** в коробочной версии — заполните маппинг:

```yaml
process:
  status_mapping:
    "1": "Новая"
    "2": "Ждёт выполнения"
    "3": "В работе"
    "4": "Ожидает"
    "5": "Завершена"
    "6": "Отложена"
    "7": "Отклонена"
    "100": "Мой кастомный статус"   # ID вашего статуса
```

**Как узнать ID кастомного статуса:**
```bash
# Перебор — посмотреть, какие статусы есть в задачах
curl -s "https://ваш-портал/rest/1/AbCdEf123456/tasks.task.list?filter[STATUS]=100&start=0" | python -m json.tool
```
Или через историю задач — в поле `STATUS` в `tasks.task.history.list`.

## Настройка ролей пользователей

Bitrix24 не имеет встроенных ролей «оператор/диспетчер/руководитель» для задач. Дашборд позволяет вручную сопоставить `user_id` → роль:

```yaml
process:
  user_roles:
    "1": "operator"      # оператор первой линии
    "5": "dispatcher"    # диспетчер
    "10": "head"         # руководитель
    "15": "executor"     # исполнитель
    "42": "support"      # служебный «Техподдержка»
```

**Роли используются для:**
- Расчёта стадий (см. раздел 14)
- Группировки в отчётах «Контроль диспетчера» и «Маршрутизация»

**Доступные роли:** `operator`, `dispatcher`, `head`, `executor`, `support`. Роль может быть любой строкой — дашборд не ограничивает набор.

**Как найти user_id:** см. раздел 6.

## Настройка обязательных полей

Дашборд проверяет заполненность обязательных полей и выводит отклонение `missing_mandatory_fields`.

```yaml
process:
  mandatory_fields:
    std_fields: ["DESCRIPTION", "DEADLINE"]
    uf_fields: ["UF_TASK_TYPE", "UF_CRM_TASK"]
```

- `std_fields` — стандартные поля Bitrix24 (`DESCRIPTION`, `DEADLINE`, `TITLE`, и т.д.)
- `uf_fields` — пользовательские поля (`UF_*`), если созданы в портале

**Как узнать имена UF_\* полей:**
```bash
curl -s "https://ваш-портал/rest/1/AbCdEf123456/tasks.task.list?select[]=*&select[]=UF_*&start=0" | python -m json.tool
```
В ответе видны все `UF_*` поля. Если поле пустое — в ответе будет `null` или ключ с `null`-значением.

## Настройка стадий процесса

Стадии — это нормативы времени, которые задача должна провести на каждой роли. Используются в отчёте «Контроль диспетчера».

```yaml
process:
  stages:
    operator_minutes: 30       # оператор должен обработать за 30 мин
    dispatcher_minutes: 120    # диспетчер — за 2 часа
    head_minutes: 360          # руководитель — за 6 часов
```

**Как это работает:**
- Время стадии = бизнес-минуты между переходами статусов, связанными с ролью
- Если задача превышает норматив — она подсвечивается в отчёте
- Нормативы можно настроить через UI (кнопка «Настройки»)

**Рекомендуемые значения:**
| Роль | Норматив | Комментарий |
|------|----------|-------------|
| Оператор | 30 мин | Первичный приём и классификация |
| Диспетчер | 2 часа | Назначение исполнителя, контроль |
| Руководитель | 6 часов | Эскалация, решение сложных вопросов |

## Первый запуск и проверка

```bash
# 1. Активировать виртуальное окружение (если есть)
source .venv/bin/activate

# 2. Установить зависимости (если ещё не установлены)
pip install -r requirements.txt

# 3. Убедиться, что config.yaml существует и заполнен (см. раздел 8)

# 4. Запустить приложение (без --demo!)
python -m sla.app

# С указанием порта:
python -m sla.app --port 8080

# С явным указанием config.yaml:
SLA_CONFIG=/path/to/config.yaml python -m sla.app
```

**Проверка после запуска:**

1. Откройте `http://localhost:8080` (или ваш порт)
2. На вкладке **«Дашборд»** должны появиться реальные задачи из портала (не демо)
3. Нажмите кнопку **«Синхронизировать»** — запустится ручная синхронизация
4. Проверьте вкладки:
   - **Отклонения** — должны быть данные по задачам
   - **Контроль диспетчера** — ежедневный чек-лист
   - **Еженедельный разбор** — задачи >7 дней, просроченные
   - **Маршрутизация** — счётчики передач

**Если данные отсутствуют:**
- Убедитесь, что `tp_user_id` правильный (см. раздел 6)
- Проверьте, что на пользователя «Техподдержка» назначены задачи
- Проверьте логи приложения (`sla_run.log`)

## Проверка конкретных API-методов

Детальные curl-примеры для каждого метода, который использует приложение:

```bash
# tasks.task.list — список задач техподдержки
curl -s "https://ваш-портал/rest/1/AbCdEf123456/tasks.task.list?filter[RESPONSIBLE_ID]=42&select[]=*&select[]=UF_*&start=0" | python -m json.tool

curl -s "https://ваш-портал/rest/1/AbCdEf123456/tasks.task.get?taskId=123" | python -m json.tool

curl -s "https://ваш-портал/rest/1/AbCdEf123456/tasks.task.history.list?taskId=123" | python -m json.tool

# task.elapseditem.getlist — учёт времени по задаче
curl -s "https://ваш-портал/rest/1/AbCdEf123456/task.elapseditem.getlist?TASKID=123" | python -m json.tool
```

## Диагностика проблем

| Симптом | Причина | Решение |
|---------|---------|---------|
| `{"error":"...","error_description":"access denied"}` | У вебхука нет прав на метод | Пересоздайте вебхук с правами `task` + `user` (см. раздел 5) |
| `{"error":"NOT_FOUND"}` | Неверный taskId | Проверьте ID задачи через `tasks.task.list` |
| Пустой дашборд (нет задач) | На tp_user_id нет назначенных задач | Назначьте задачи на пользователя «Техподдержка» или измените `tp_user_id` |
| Комментарии не загружаются | Метод deprecated на новых порталах (после 2023) | Переключите сигнал первого ответа на «Смена статуса» |
| Медленная синхронизация | Много задач, большой batch_size | Уменьшите `batch_size` до 20–30, увеличьте `request_timeout` до 60 |
| `ConnectionError` | Нет сетевого доступа до портала | Проверьте HTTPS-доступ, настройки прокси, firewall, DNS |
| `SSLError` / `CERTIFICATE_VERIFY_FAILED` | Самоподписанный сертификат на коробке | Установите корневой сертификат в ОС или настройте Let's Encrypt |
| Статусы отображаются числами | Не заполнен `status_mapping` | Заполните `process.status_mapping` в config.yaml (см. раздел 11) |
| Отклонения пусты | Нет синхронизированных задач или слишком строгие пороги | Нажмите «Синхронизировать», ослабьте пороги в `process.deviations` |
| Роли не работают | Не заполнен `user_roles` | Заполните `process.user_roles` в config.yaml (см. раздел 12) |
| Ошибка «config not found» | Файл config.yaml не существует | Создайте config.yaml по шаблону из раздела 8 |

## REST API методы, используемые приложением

| Метод | Назначение | Параметры | Когда вызывается |
|-------|-----------|-----------|------------------|
| `tasks.task.list` | Список задач техподдержки | `filter[RESPONSIBLE_ID]`, `select[]=*,UF_*`, `start` | Каждая синхронизация (постранично) |
| `tasks.task.get` | Полные данные одной задачи | `taskId` | При обновлении конкретной задачи |
| `tasks.task.history.list` | История изменений (статусы, переносы дедлайна, назначения) | `taskId` | При обновлении каждой задачи |
| `task.elapseditem.getlist` | Записи учёта времени | `TASKID` | При обновлении каждой задачи |
| `task.commentitem.getlist` | Комментарии к задаче | `TASKID` | При обновлении (может быть deprecated) |
| `user.get` | Информация о пользователях | `ID` или `FILTER[NAME]` | При инициализации и синхронизации |

**Частота вызовов при синхронизации:**
- 1 вызов `tasks.task.list` на каждые `batch_size` задач (по умолчанию 50)
- Для каждой задачи: 1× `tasks.task.get` + 1× `tasks.task.history.list` + 1× `task.elapseditem.getlist` + 1× `task.commentitem.getlist`
- Для N задач = 1 + N×4 вызовов API

## Безопасность и права доступа

- **Вебхук = пароль.** URL вебхука даёт полный доступ к API с правами владельца. Не публикуйте его.
- **config.yaml в .gitignore.** Токен доступа не попадёт в репозиторий.
- **Создавайте вебхук от администратора** — иначе REST вернёт только задачи самого владельца.
- **Ограничьте сетевой доступ** к дашборду:
  - Используйте firewall (разрешить доступ только с IP администраторов)
  - Настройте reverse proxy (nginx, Caddy) с базовой аутентификацией
  - Не открывайте порт 8080 в интернет без пароля
- **Регулярно ротируйте `webhook_token`**: пересоздайте входящий вебхук → обновите `config.yaml`
- **Логи приложения** (`sla_run.log`) могут содержать названия задач и ID пользователей — защитите файл лога (права `600`)

## Частые сценарии

### «Хочу добавить ещё одного исполнителя техподдержки»

На данный момент дашборд поддерживает фильтрацию по одному `tp_user_id`. Решения:
- Назначать все задачи ТП на служебного пользователя «Техподдержка»
- Вручную указать роли в `user_roles` — дашборд сгруппирует задачи по ответственному

### «Хочу фильтровать по группе задач»

Используйте фильтр в UI (поле «Группа»). `GROUP_ID` берётся из задачи. Группы автоматически подгружаются из портала при синхронизации.

### «Хочу 24/7 SLA для критичных задач»

Пока не реализовано разделение по приоритетам. Обход:
```yaml
working_hours:
  workdays: [0, 1, 2, 3, 4, 5, 6]  # все дни недели
  start: "00:00"
  end: "23:59"                       # круглосуточно
```
Это повлияет на все приоритеты, а не только на критичные.

### «Хочу экспорт в Excel»

Пока не реализовано. Используйте API-эндпоинты дашборда:
- `/api/tasks` — список задач с метриками (JSON)
- `/api/deviations` — отклонения (JSON)
- `/api/summary` — сводка KPI (JSON)

Конвертируйте JSON в Excel через скрипт или онлайн-инструменты.

### «Комментарии не загружаются»

На новых порталах (после 2023 года) комментарии задач перенесены в Живую ленту (`log`). REST-метод `task.commentitem.getlist` может вернуть пустой результат или ошибку. **Решение:** переключите сигнал первого ответа на «Смена статуса» — это самый надёжный метод.

### «Самоподписанный сертификат»

В коробочной версии Bitrix24 часто используется самоподписанный SSL-сертификат. Решения (в порядке предпочтения):
1. **Установите корневой сертификат организации** в доверенные на сервере дашборда
2. **Настройте Let's Encrypt** на портале (Certbot для Linux)
3. **Добавьте поддержку `verify_ssl: false`** в config.yaml (пока не реализовано в коде — потребуется доработка `sla/bitrix.py`)

### «Хочу изменить частоту синхронизации»

Измените `sync.interval_minutes` в config.yaml:
```yaml
sync:
  interval_minutes: 30  # каждые 30 минут
```

### «Хочу несколько tp_user_id»

Текущая версия поддерживает только один `tp_user_id`. Для добавления нескольких потребуется доработка `sla/sync.py`.
